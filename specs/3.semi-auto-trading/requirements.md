# semi-auto-trading — 需求规格

## 概述

将 Alpha 定投信号转化为真实交易执行：用户绑定 Binance API Key，查看信号后"一键确认下单"（半自动），每笔订单自动记录供后续复盘。

## 项目信息

- 项目名: trade-pilot
- 架构类型: pnpm monorepo (Turborepo) — React + Vite 前端 / Hono + tRPC 后端 / Drizzle + PostgreSQL

## 需求版本

| 日期       | 版本 | 说明     |
| ---------- | ---- | -------- |
| 2026-06-16 | v1   | 初始需求 |

## 用户故事

- 作为个人交易者，我想要安全地绑定我的 Binance API Key，以便 APP 可以代我执行下单，同时确保我的资产不会被非预期操作。
- 作为个人交易者，我想要在下单前看到明确的参数预览（币种、方向、金额），并通过二次确认后才执行，以避免误操作导致损失。
- 作为个人交易者，我想要每笔下单结果（成功/失败）都被记录，以便后续复盘。

## 功能需求

1. [F-001] **API Key 绑定**：用户在设置页输入 Binance API Key + Secret，系统校验有效性（仅验证读取权限即可），加密存储到 DB。
2. [F-002] **权限校验提示**：绑定时检查 API Key 是否具备现货交易权限，若无则提示用户开启（展示 Binance 文档链接），但不强制阻止保存。
3. [F-003] **下单参数配置**：用户在 Alpha 信号页或下单弹窗中设置本次下单参数：交易标的（从信号自动填入）、方向（买入/做多，MVP 仅支持现货买入）、金额（USDT）。
4. [F-004] **下单预览 + 二次确认**：展示下单摘要弹窗（标的、方向、金额、预估手续费、当前价格），用户点击"确认下单"才真正执行。
5. [F-005] **订单执行**：通过 Binance REST API（现货市价单 `POST /api/v3/order`）提交订单，使用用户绑定的 API Key。
6. [F-006] **执行结果记录**：订单成功/失败均记录到 `trade_orders` 表，含 Binance 返回的订单 ID、成交价格、手续费、状态。
7. [F-007] **失败反馈**：下单失败时，前端展示错误原因（余额不足、权限不够、网络错误等）。
8. [F-008] **全局风险免责**：设置页和下单弹窗均展示风险声明："此操作使用真实资金，所有交易由您自行负责，本产品不承担任何投资损失。"

## 非功能需求

- **安全**：API Key 和 Secret 使用 AES-256-GCM 加密存储，加密密钥来自服务端环境变量（不入 DB）；前端不接触 API Key 明文。
- **最小权限**：仅申请"现货交易"权限，MVP 不支持合约；绑定文档中明确提示用户不要开启提现权限。
- **幂等性**：下单接口前端防抖 + 后端校验（同一请求 ID 不重复下单）。

## 验收标准

- [ ] [AC-001] 用户可在设置页输入并保存 Binance API Key，保存后显示"已绑定（已脱敏展示末4位）"。
- [ ] [AC-002] 点击 Alpha 信号上的"下单"按钮，弹出包含标的、金额、当前价的预览弹窗。
- [ ] [AC-003] 点击"确认下单"后，Binance 返回成功时，前端显示"下单成功，订单 ID: xxx"。
- [ ] [AC-004] 下单成功后，记录写入 DB，可在订单列表（Feature 4）中查到。
- [ ] [AC-005] 下单失败时，前端显示具体错误原因，不卡死。
- [ ] [AC-006] 未绑定 API Key 时，"下单"按钮显示"请先绑定 Binance API Key"并跳转设置页。

## 依赖

- Binance REST API（`POST /api/v3/order`，需 API Key + HMAC-SHA256 签名）
- Feature 2（alpha-dca-strategy）中的信号作为下单入口
- Feature 4（order-review）中的 `trade_orders` 表（本 feature 负责写入）

## 开放问题

- MVP 仅支持现货市价买入。合约、限价单、止盈止损等进入后续迭代。
- 加密密钥管理：MVP 使用服务端环境变量（`ENCRYPTION_KEY`），后续考虑 KMS 或 HashiCorp Vault。
